=== STEP 0: ALWAYS START WITH 4 KPI WIDGETS ===
Before adding any charts or tables, always include exactly 4 KPI widgets at the top of the dashboard (y=0, x=0/3/6/9).
Choose 4 meaningful KPIs based on the data source and user request. Common KPIs:
- "Total [Records]"         => valueKey="count"
- "Open [Records]"          => valueKey="count" with a filter condition (or a relevant status-filtered count)
- "Closed [Records]"        => valueKey="count"
- "Average Score"           => valueKey="Score" (only if Score column exists, otherwise pick another count-based KPI)
If you cannot find 4 distinct meaningful KPIs, repeat count-based KPIs with different labels (e.g. "Total Hazards", "Open Hazards", "Closed Hazards", "In Progress").

=== STEP 1: DECIDE WIDGET TYPE FROM THE TITLE ===
Look at the widget title and pick type + chartType:
- Title contains "by [Category]" or "per [Category]"  => type=chart, chartType=bar
- Title contains "over time" or "trend"               => type=chart, chartType=line
- Title contains "distribution" or "breakdown"        => type=chart, chartType=pie
- Title contains "total", "count", "average", "avg" (single number) => type=kpi
- Title contains "detail", "list", "report"           => type=table
- Title contains "map" or "location"                  => type=map
- Title contains "gauge", "compliance rate", "% complete", "risk score" => type=gauge
- Title contains "vs last", "compared to", "change", "delta"           => type=stat
- Title contains "progress", "ranked", "top [N] by"                    => type=progress
- Title contains "donut", "share of", "proportion"                     => type=donut
- Title contains "heatmap" or two-axis matrix (e.g. "Status by Department") => type=heatmap

=== STEP 2: POPULATE config BASED ON TYPE ===

--- CHART: config MUST contain xKey + aggregation ---
Look at the title to pick xKey. Match the "by [X]" phrase to the column name from the data source:
  "by Type"        => xKey = HazardType  (or the column whose name contains "Type")
  "by Department"  => xKey = Department
  "by Status"      => xKey = Status
  "by Location"    => xKey = Location
  "by Programme"   => xKey = Programme
  "by Sub-Type"    => xKey = SubType
  "over Time"      => xKey = StartDt   (or the date column)
  "by Person"      => xKey = PersonResponsible

For aggregation:
  - Default to aggregation="count" (count records per group, no yKey needed)
  - Use aggregation="avg" + yKey="Score" ONLY when title mentions "score" or "risk"
    AND "Score" exists in the dataSource columns list
  - Use aggregation="sum" + yKey when title mentions "total [numeric column]"
    AND that column exists in the dataSource columns list

For time-series charts (xKey is a date column), add dateGroup to config:
  "monthly"        => dateGroup = "monthly"        (groups by Mon YYYY, e.g. "Apr 2026")
  "quarterly"      => dateGroup = "quarterly"      (groups by Q1 2026, Q2 2026, ...)
  "yearly"         => dateGroup = "yearly"         (groups by 2024, 2025, 2026)
  "financial year" => dateGroup = "financial_year" (groups by FY2025-26, FY2026-27)
  If no grouping is specified for a date axis, omit dateGroup.

--- KPI: config MUST contain valueKey ---
  "Total [records]"    => valueKey = "count", format = "number"
  "Average/Avg Score"  => valueKey = "Score",  format = "number"  (ONLY if Score exists in columns)
  "Open [records]"     => valueKey = "count",  format = "number"

  For any KPI that counts records use valueKey = "count".
  For any KPI that needs a real column ensure that column is in the dataSource columns list.

  WRONG: valueKey="Score" when "Score" does NOT appear in the dataSource columns list.
  RIGHT: valueKey="count" when no numeric aggregation column is available.

  COUNTING UNIQUE VALUES — use aggregation + aggregateColumn:
  - "Total Departments Involved" => aggregation="count_distinct", aggregateColumn="Department", valueKey="count"
  - "Total Locations Assessed"   => aggregation="count_distinct", aggregateColumn="Location",   valueKey="count"
  - "Total Hazard Types"         => aggregation="count_distinct", aggregateColumn="HazardType",  valueKey="count"
  Use count_distinct whenever the KPI title says "unique", "distinct", "different", or counts a category noun.
  count_distinct requires aggregateColumn to be a real column in the dataSource columns list.

  KPI PRE-FILTERS — to count a subset of records, add a "*Filter" config key:
  The key name is the column name with "Filter" appended (camelCase).
  The value can be a single string OR a comma-separated list (generates SQL IN):
    statusFilter = "Open"                        → WHERE Status = 'Open'
    statusFilter = "In Progress,Not Started"      → WHERE Status IN ('In Progress','Not Started')
    typeFilter   = "Hazard"                       → WHERE Type = 'Hazard'
  Use this instead of inventing a separate dataSource or filter for KPI subsets.

  HOW TO PICK statusFilter VALUES — always use "knownValues" from the column metadata:
  Each column may have a "knownValues" array with the real distinct values from the database.
  When the user wants an "Open" or "Active" KPI, DO NOT guess "Open" — instead:
    1. Look at Status knownValues (e.g. ["Closed","In Progress","Not Started","Overdue"])
    2. Identify TERMINAL statuses: any value whose name suggests completion or rejection,
       such as: Closed, Completed, Cancelled, Archived, Resolved, Done, Rejected, Expired
    3. "Open" KPI  => statusFilter = all knownValues EXCEPT terminal ones (comma-separated)
    4. "Closed" KPI => statusFilter = only the terminal knownValues (comma-separated)
    5. If knownValues is absent, fall back to statusFilter = "Open" or "Closed" as a best guess.
  Example — if knownValues = ["Closed","In Progress","Not Started","Overdue"]:
    "Open Risks" KPI   => statusFilter = "In Progress,Not Started,Overdue"
    "Closed Risks" KPI => statusFilter = "Closed"

--- GAUGE: config MUST contain valueKey, max ---
Shows a half-doughnut gauge of a single scalar value within a min–max range.
  valueKey    — "count" or a numeric column (same rules as KPI valueKey)
  min         — scale minimum, usually 0
  max         — scale maximum (e.g. "100" for a percentage, total record count, etc.)
  format      — "number" | "percent" | "currency"
  color       — arc colour (same palette as KPI)
  label       — text shown under the number (falls back to widget title)

--- STAT: config MUST contain valueKey + prevValue ---
Like a KPI but displays a delta arrow + percentage change vs a reference value.
  valueKey      — "count" or a numeric column (same rules as KPI)
  prevValue     — hardcoded comparison number as a STRING (e.g. "142") representing the previous period value
  deltaLabel    — context string (e.g. "vs last quarter", "vs last year")
  goodDirection — "up" (increase is good, shown green) | "down" (decrease is good, shown green)
  format / color / prefix / suffix — same as KPI

Estimate prevValue from context or set it to "0" if unknown.

POSITION RULES (12-column grid, no overlaps):

  WRONG:   { "id": "w1", "x": 0, "y": 0, "w": 3, "h": 2, ... }

- Chart:    w=6, h=4
- Table:    w=12, h=5
- Map:      w=6, h=5
- Gauge:    w=3, h=3
- Stat:     w=3, h=2
- Progress: w=6, h=4
- Donut:    w=6, h=4

Layout:

- Charts go in the next row (y=2). Two charts fit per row: x=0 and x=6. If 3 charts, use a second chart row.
- Tables always use x=0, w=12, placed below all charts.
- Maps use w=6, placed like charts (x=0 or x=6).
- Never place two widgets with the same y and overlapping x ranges.
- Always use the FULL 12-column width — do NOT leave all widgets at x=0.
